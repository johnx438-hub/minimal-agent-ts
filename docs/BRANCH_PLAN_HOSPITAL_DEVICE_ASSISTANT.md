# 分支计划：医护人员设备端办公辅助（记录稿）

> **状态**: 仅记录，**不进** `ROADMAP.md`，**不改**主仓内核。  
> **来源**: 2026-07 医院侧对「医护人员设备端办公辅助」潜力的询问。  
> **主仓定位不变**: `minimal-agent-ts` 仍为 session 内上下文实验 + 可读 harness。

---

## 1. 询问在问什么（理解）

对方兴趣点偏向 **办公辅助**，而非完整临床信息系统：

| 更可能 | 较不可能（需单独澄清） |
|--------|------------------------|
| 交接班摘要、会议/培训纪要 | 正式 EMR 录入与医嘱闭环 |
| 制度/流程/操作规范检索与问答 | 跨科室完整病历读写 |
| 设备旁快速查资料、填表草稿 | 监管级审计与等保全套 |
| 排班/通知类信息的整理辅助 | 替代 HIS/LIS/PACS |

**设备端**通常指：科室 PC、移动查房平板、护士站终端、院内网环境——强调 **内网、短任务、只读或弱写入**，不是公开互联网漫游。

## 2. 与主仓的关系

```
minimal-agent-ts（主仓，冻结为 harness）
        │
        │  fork / 配置轨（本计划）
        ▼
hospital-device-assistant（工作名，未定正式 repo 名）
  - 继承：ReAct、context、workflow、spawn、TUI、Skills、MCP
  - 剥离/默认关闭：run_shell、宽域 web_fetch、任意路径 write
  - 新增：院内工具适配层（只读 API、受控模板填写）
```

**原则**: 主仓继续实验；医院线 **只在外层分支** 做预设、策略与集成，避免把合规约束写进通用 `agent.ts`。

---

## 3. 适配度（简评）

| 维度 | 判断 |
|------|------|
| 长文档 / 多轮整理 | 主仓 pointerize + recall **可用** |
| 流程化（起草→核对） | workflow + spawn **可用** |
| 设备端轻办公 | **方向匹配**，需窄工具面 |
| 病例管理系统 | **不匹配**；若院方要的是 EMR，应另立项 |
| 合规与审计 | 主仓 **不具备**；分支或院内部署层必须补 |

---

## 4. 分支产品边界（对外口径）

**可做（MVP 叙事）**

- 制度 / 指南 / 宣教材料的 **检索与摘要**（接院内知识库或 MemFileCli）
- 交接班、晨会、培训的 **纪要草稿**（人工确认后粘贴到现有系统）
- 表单 / 表格的 **填写建议**（不自动提交 HIS）
- 专科 **只读** 子任务（spawn 预设：文献摘要、操作步骤核对清单）

**明确不做（除非单独立项）**

- 诊断建议、用药决策（避免「医疗 AI 决策」合规雷区）
- 直接写入患者标识数据到本地明文文件
- 默认开放 shell、外网随意抓取
- 替代院方已有 HIS / EMR / 护理文书系统

## 5. 技术分支草案（未实施）

### 5.1 建议 repo / 分支形态

| 选项 | 说明 |
|------|------|
| `minimal-agent-ts-hospital` fork | 独立 `agent.json` + `agents/` + 禁用工具列表 |
| 主仓 `hospital/` 配置目录 only | 改动最小，但合规策略易污染主仓——**次选** |

推荐：**独立 fork**，定期 cherry-pick 主仓轨 B / 事件 / TUI 修复。

### 5.2 默认策略（Lite + 院内）

```bash
ENABLE_ZVEC=0
ENABLE_EMBEDDINGS=0
ALLOW_SHELL=0
ALLOW_WEB=0          # 或仅院内域名白名单
MAX_TURNS=30         # 可按院方调整
```

`builtin_tools` 建议仅保留：`read_file`（沙箱目录）、`grep_search`、`recall_query`、`invoke_skill` + **院内 MCP 工具**。

### 5.3 集成面（待院方确认）

- [ ] 知识库：MemFileCli / 院内 Wiki API / 仅文件共享目录
- [ ] 身份：AD / 钉钉 / 企业微信 / 院自建 SSO（**不在 harness 内实现**）
- [ ] 部署：院内 VM / 容器；**不出院** 调用公网 LLM 是否允许
- [ ] 模型：私有化部署 vs 院方已采购 API
- [ ] 审计：谁问了什么、引用了哪条制度——需 **旁路日志服务**

### 5.4 MVP 里程碑（若立项）

| 阶段 | 交付 | 周期（粗估） |
|------|------|--------------|
| P0 演示 | 只读问答 + 制度摘要 + TUI，假数据知识库 | 2–4 周 |
| P1 试点 | 院内知识库 MCP + 审计日志 + 账号对接 | 1–2 月 |
| P2 扩展 | workflow（纪要→核对）、科室 spawn 预设 | 视试点反馈 |

## 6. 风险与院方需澄清的问题

**风险**

- 院方口头「办公辅助」实际期望可能是 **护理文书 / 病程录入**
- 设备端环境差异大（Windows 老版本、无 Node、离线）
- LLM 幻觉在医疗场景 **舆论与合规** 敏感——即使不做诊断

**建议回访问题清单**

1. 主要场景 Top3？（交接班 / 制度查 / 培训 / 其他）
2. 是否必须 **离网或私有化模型**？
3. 能否接受 **人工确认后复制** 到现有系统，而非自动写回？
4. 已有知识库形态？（PDF 共享盘 / Wiki / HIS 附件）
5. 等保 / 个人信息保护 **院方负责还是供应商负责**？
6. 试点科室与设备类型？（护士站 PC / Android 平板）
